# AI エージェント開発フレームワーク検証 -- 評価フォーマット集

## 1. 概要

### 目的

本フォーマット集は、AI エージェントを活用した開発フレームワーク（Salesforce Multi-Agent, Cursor, Windsurf, Devin, Claude Code, GitHub Agent など）を体系的に検証・評価し、組織の標準フレームワークを選定するための統一評価基盤を提供する。

### 設計思想: 二層構造（コア + ドメイン拡張）

本フォーマット集は **二層構造** で設計されている。

- **コア層**: フレームワーク共通の評価項目。ツールやドメインに依存せず、あらゆる AI エージェントフレームワークに適用可能
- **ドメイン拡張層**: 特定のドメイン（Salesforce、Web フロントエンド、データパイプラインなど）に固有の評価項目。コア層を補完する形で追加する

この分離により、以下を実現する:

- 異なるフレームワーク間の **公平な比較**（コア層で統一基準）
- ドメイン固有の要件への **柔軟な対応**（拡張層で個別対応）
- 新規ドメインへの **容易な展開**（拡張テンプレートの追加のみ）

### 対象読者

| 読者 | 主な利用シーン |
|:-----|:---------------|
| フレームワーク構築者 | プロファイルの記録、構成要素の設計改善 |
| 検証実行者 | 各フェーズ・案件の評価記入 |
| 比較評価者 | 複数フレームワークの横断比較 |
| プロジェクトリーダー | 標準化判定、意思決定 |

## 3. 使い方フロー

### 3.1 初回セットアップ

1. `00_framework_profile.md` をコピーし、評価対象フレームワークの構成を記録する
2. 必要に応じてドメイン拡張ファイル（`ext_sf_*.md` など）を準備する
3. `04_trend_tracker.md` をコピーし、推移記録の器を用意する

### 3.2 各検証実行時

1. `01_phase_eval_template.md` をコピーし、各フェーズ完了時に評価を記入する
   - ドメイン固有の評価が必要な場合、対応する `ext_*` ファイルも併せて記入する
2. 案件全体の完了時に `02_agent_capability_eval.md` を記入する
3. `04_trend_tracker.md` に当該検証回のメトリクスを追記する

### 3.3 比較・判定時

1. 複数フレームワークの検証が完了したら `03_cross_framework_comparison.md` を記入する
2. 同一フレームワークで 3 回以上の検証が完了したら `05_standardization_scorecard.md` で標準化判定を実施する

### フロー図

```
[初回セットアップ]
  00_framework_profile.md を記入
          |
[各検証実行]  (検証ごとに繰り返し)
  01_phase_eval_template.md  +  ext_sf_*.md を記入
          |
  02_agent_capability_eval.md を記入
          |
  04_trend_tracker.md に追記
          |
[比較・判定]  (条件を満たしたら)
  03_cross_framework_comparison.md を記入
          |
  05_standardization_scorecard.md で標準化判定
```

## 5. 新しいドメイン拡張の追加方法

### 5.1 命名規則

```
ext_{domain}_phase{N}_{name}.md
```

| 要素 | 説明 | 例 |
|:-----|:-----|:---|
| `{domain}` | ドメイン識別子（2〜4 文字の略称） | `sf`（Salesforce）, `web`（Web フロントエンド）, `data`（データパイプライン） |
| `{N}` | 対応する汎用フェーズ番号（0〜5） | `0`（セットアップ）, `1`（要件分析） |
| `{name}` | 評価内容の簡潔な名称 | `setup`, `requirements`, `deploy` |

### 5.2 設計原則

1. **コア層と重複しない**: `01_phase_eval_template.md` で既に評価される項目は含めない
2. **ドメイン固有の観点に集中する**: そのドメインでなければ評価できない項目のみを追加する
3. **統一評価スケールを使用する**: コア層と同じ 5 段階評価（1〜5）および成熟度 4 段階（L1〜L4）を使用する
4. **コア層からの参照を可能にする**: `01_phase_eval_template.md` のセクション 6（ドメイン拡張セクション）からリンクできる構造にする

### 5.3 作成手順

1. 既存の拡張ファイル（例: `ext_sf_phase0_setup.md`）を参考にテンプレートを作成する
2. ヘッダーにフレームワークプロファイル参照と対応する汎用フェーズを明記する
3. ドメイン固有の評価項目を設計する
4. 本 README のファイル一覧（セクション 2）に追加する

## 6. 統一評価スケール

### 6.1 5 段階評価（品質評価用）

| スコア | ラベル | 定義 |
|:-----:|:-------|:-----|
| 5 | 優秀 | 人間の成果物と同等以上。修正不要 |
| 4 | 良好 | 軽微な修正（5 分以内）で利用可能 |
| 3 | 合格 | 修正が必要だが方向性は正しい（30 分以内で修正可能） |
| 2 | 不合格 | 大幅な修正が必要（やり直しに近い） |
| 1 | 失敗 | 全くの的外れ、または出力不能 |

### 6.2 成熟度 4 段階（標準化判定用）

| レベル | ラベル | 定義 |
|:------:|:-------|:-----|
| L1 | Initial（初期） | まだ安定していない |
| L2 | Developing（発展中） | 改善傾向にあるが変更が必要 |
| L3 | Defined（確立） | 安定して機能、軽微な調整のみ |
| L4 | Optimized（最適化） | 十分に機能、改善余地が小さい |

### 6.3 特殊値の使用ルール

| 値 | 意味 | 使用場面 |
|:---|:-----|:---------|
| N/A | 該当なし（Not Applicable） | その評価項目が対象フレームワーク・案件に該当しない場合。スコア集計から除外する |
| N/M | 未測定（Not Measured） | 測定すべきだが今回は測定できなかった場合。次回検証での測定を必須とする |

## 7. FAQ

### Q1: コア層だけで評価は十分か？

**A**: コア層のみでフレームワーク間の比較・標準化判定に必要な評価は完結する。ただし、ドメイン固有の品質要件（例: Salesforce のガバナ制限準拠、Web のアクセシビリティ準拠など）を評価するにはドメイン拡張が必要。まずコア層で評価を開始し、不足を感じた時点でドメイン拡張を追加する運用を推奨する。

### Q2: 他チームと比較する場合の最低限のフォーマットは？

**A**: 以下の 4 ファイルが最低限必要:

1. `00_framework_profile.md` -- 比較対象の構成を揃えて記録
2. `01_phase_eval_template.md` -- 統一基準でのフェーズ評価
3. `02_agent_capability_eval.md` -- エージェント能力の横断比較
4. `03_cross_framework_comparison.md` -- 比較結果の集約

### Q3: ドメイン拡張を新規作成する場合の手順は？

**A**: セクション 5「新しいドメイン拡張の追加方法」を参照。既存の `ext_sf_*.md` をテンプレートとして流用し、ドメイン固有の評価項目に書き換えることで効率的に作成できる。

### Q4: 検証何回目から標準化判定を行えるか？

**A**: 最低 3 回の検証完了が必要。さらに、案件種別が 2 種以上（新規開発・機能追加・バグ修正のうち 2 つ以上）含まれていることが前提条件となる。詳細は `05_standardization_scorecard.md` のセクション 1「標準化判定の前提条件」を参照。

### Q5: 04_trend_tracker.md が存在しない場合はどうするか？

**A**: `04_trend_tracker.md` は各検証完了時に追記する累積型のファイル。初回セットアップ時にテンプレートをコピーして用意し、検証ごとにメトリクスを追記していく。標準化判定（`05_standardization_scorecard.md`）ではこのファイルのデータを集約して使用する。

### 関係の説明

| 関係 | 説明 |
|:-----|:-----|
| `00` -> `01` | フェーズ評価時にフレームワーク構成を参照する |
| `01` <-> `ext_*` | コア評価とドメイン拡張評価は同一フェーズで相互補完する |
| `01` -> `02` | 各フェーズの評価結果を案件単位のエージェント能力評価に集約する |
| `02` -> `03` | 複数フレームワークの能力評価を横断比較する |
| `02` -> `04` | 各検証回の定量メトリクスを推移記録に蓄積する |
| `03` -> `05` | 比較結果を標準化判定の入力とする |
| `04` -> `05` | 推移データを標準化判定の定量根拠とする |
