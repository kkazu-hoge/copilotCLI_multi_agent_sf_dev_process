# エラーハンドリング方針（Error Handling Policy）

> 最終更新: 2026-03-09
> 管理者: {チーム名}
> ステータス: 承認済み

---

## 1. 目的

Salesforce 組織における統一的なエラーハンドリング方針を定義し、障害の迅速な検知・原因特定・復旧を実現する。本ドキュメントは全案件に適用される。

---

## 2. エラー分類

### 2.1 エラーカテゴリ

| カテゴリ | 例 | 対処方針 | ユーザーへの表示 |
|---------|----|---------|--------------  |
| 業務エラー | バリデーション不合格、必須項目未入力、ステータス遷移不正 | ユーザーに修正を促すメッセージを表示 | 具体的なガイダンスを表示 |
| システムエラー | ガバナ制限超過、ヌルポインタ、型変換エラー | ログ記録 + 管理者通知 + ユーザーに一般的なエラーメッセージ表示 | 「予期しないエラーが発生しました。管理者にお問い合わせください。」 |
| 連携エラー | 外部 API タイムアウト、認証失敗、レスポンス不正 | リトライ + ログ記録 + 管理者通知 | 「外部システムとの通信に失敗しました。しばらく経ってから再度お試しください。」 |
| データ整合性エラー | 重複レコード、参照先レコード不在、循環参照 | ログ記録 + データ修復手順の提示 | 「データの不整合が検出されました。管理者にお問い合わせください。」 |

## 3. 例外ハンドリングの実装方針

### 3.1 基本原則

| 原則 | 内容 |
|------|------|
| 早期検出 | エラーは可能な限り早い段階（入力バリデーション、事前条件チェック）で検出する |
| 適切な粒度 | try-catch は必要最小限の範囲で使用する。トランザクション全体を包む巨大な try-catch は禁止 |
| 情報の保持 | 例外をキャッチして再スローする場合は、元の例外情報を保持する |
| サイレント失敗の禁止 | 空の catch ブロックは禁止。最低限ログ記録を行うこと |

### 3.2 カスタム例外クラスの設計

プロジェクト全体で統一したカスタム例外クラス階層を使用する。

```
ApplicationException（基底クラス）
├── BusinessException          # 業務エラー（ユーザーにメッセージ表示）
│   ├── ValidationException    # バリデーションエラー
│   └── StateTransitionException  # ステータス遷移エラー
├── IntegrationException       # 外部連携エラー
│   ├── CalloutException       # コールアウト失敗
│   └── DataMappingException   # データ変換エラー
└── DataIntegrityException     # データ整合性エラー
```

### 3.3 例外処理の層別方針

| レイヤー | 例外処理の方針 |
|---------|-------------|
| Controller / LWC Apex | try-catch で BusinessException を捕捉し、ユーザー向けメッセージを返却。SystemException は ErrorLogger に記録後、汎用メッセージを返却 |
| Service | ビジネスロジックの検証で BusinessException をスロー。予期しない例外は catch してログ記録後に再スロー |
| Domain / Selector | 例外は原則としてスローのみ（catch しない）。呼び出し元で処理する |
| Trigger Handler | Service 層の例外を catch し、`addError()` でレコードにエラーを付与 |
| Batch / Queueable | 各レコードの処理を個別に try-catch し、エラーレコードを記録。バッチ全体の中断を防ぐ |

## 4. ログ記録の方針

### 4.1 ログ記録の仕組み

エラーログは `Application_Log__c` カスタムオブジェクトに記録する。`System.debug()` のみに依存したエラーログは禁止する（デバッグログは一時的であり、永続化されないため）。

### 4.2 ログ記録の必須項目

| 項目 | 内容 | 必須 |
|------|------|------|
| Log_Level__c | ERROR / WARN / INFO | ○ |
| Category__c | BUSINESS / SYSTEM / INTEGRATION / DATA_INTEGRITY | ○ |
| Class_Name__c | 例外が発生した Apex クラス名 | ○ |
| Method_Name__c | 例外が発生したメソッド名 | ○ |
| Message__c | エラーメッセージ | ○ |
| Stack_Trace__c | スタックトレース（Long Text Area） | ○（SYSTEM / INTEGRATION のみ） |
| Record_Id__c | 関連レコード ID | △ |
| User_Id__c | 実行ユーザー ID | ○ |
| Timestamp__c | 発生日時（UTC） | ○ |
| Context__c | 追加コンテキスト情報（JSON） | △ |

### 4.3 ErrorLogger ユーティリティ

ログ記録は統一された `ErrorLogger` ユーティリティクラスを経由して行う。直接 `Application_Log__c` を insert するコードは禁止する。

```
ErrorLogger の主要メソッド:
- ErrorLogger.logError(Exception ex, String className, String methodName)
- ErrorLogger.logError(Exception ex, String className, String methodName, Id recordId)
- ErrorLogger.logWarning(String message, String className, String methodName)
- ErrorLogger.logIntegrationError(Exception ex, String integrationName, String endpoint)
```

### 4.4 ログ記録のパフォーマンス考慮

| 方針 | 内容 |
|------|------|
| 非同期ログ記録 | 通常時は `Platform Event`（`Log_Event__e`）を発行し、サブスクライバーでログを永続化する。これによりDML 回数のガバナ制限を消費しない |
| フォールバック | Platform Event の発行に失敗した場合は、直接 `Application_Log__c` に insert する |
| ログの保持期間 | ERROR: 90日、WARN: 30日、INFO: 14日。期限切れログは日次バッチで削除 |

## 5. ユーザーへのエラー表示

### 5.1 メッセージ設計の原則

| 原則 | 内容 |
|------|------|
| 明確さ | 何が起きたかをユーザーが理解できる言葉で説明する |
| 行動指示 | ユーザーが次に何をすべきかを具体的に示す |
| 技術情報の非表示 | スタックトレース、クラス名、Apex 固有のエラーメッセージはユーザーに表示しない |
| 多言語対応 | エラーメッセージはカスタムラベルで管理し、多言語化に対応する |

### 5.2 LWC でのエラー表示パターン

| エラー種別 | 表示方式 | 用途 |
|-----------|---------|------|
| 項目バリデーション | インライン（項目直下に赤文字） | 入力フォームの項目レベルエラー |
| フォーム全体のエラー | トースト（error タイプ） | 保存失敗、一般的な業務エラー |
| 警告 | トースト（warning タイプ） | 処理は完了したが注意が必要な場合 |
| システムエラー | エラーパネル + リファレンス番号 | 予期しないエラー（ログの参照番号を表示） |

## 6. トランザクション制御

### 6.1 Savepoint の使用方針

| シナリオ | Savepoint の使用 | 備考 |
|---------|-----------------|------|
| 複数オブジェクトの一括更新 | 推奨 | 部分的な成功によるデータ不整合を防ぐ |
| コールアウト前の DML | 必須 | コールアウト失敗時にロールバックするため |
| バッチ処理の個別レコード | 非推奨 | `Database.insert(records, false)` で部分成功を許容する |

### 6.2 部分成功の方針

| 処理種別 | 部分成功の許容 | 理由 |
|---------|-------------|------|
| ユーザー操作（画面） | 非許容（全件成功 or 全件ロールバック） | ユーザーが結果を明確に把握できるようにする |
| バッチ処理 | 許容 | 1件の失敗で全体が停止するのを防ぐ |
| 外部連携（Inbound） | 許容 | エラーレコードのみ再送可能にする |

## 7. 本ドキュメントの参照方法

### サブエージェントへのガイダンス

- **sf-designer**: 設計書にエラーハンドリング方式（業務エラー/システムエラーの区別、ログ方針）を記載すること
- **sf-implementer**: カスタム例外クラス階層、ErrorLogger の使用、Platform Event によるログ記録を本方針に従って実装すること
- **sf-tester**: 異常系テスト（バリデーション違反、ガバナ制限、コールアウト失敗）を必ず実施し、エラーメッセージとログ記録を検証すること
- **sf-code-reviewer**: 空の catch ブロック、直接的な `Application_Log__c` の insert、技術情報のユーザー表示がないかをチェックすること
