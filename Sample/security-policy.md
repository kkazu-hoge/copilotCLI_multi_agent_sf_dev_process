# セキュリティ方針（Security Policy）

> 最終更新: 2026-03-09
> 管理者: {チーム名}
> ステータス: 承認済み

---

## 1. 目的

Salesforce 組織におけるデータセキュリティとアクセス制御の統一方針を定義する。本ドキュメントは全案件に適用され、Salesforce のセキュリティモデル（組織レベル → オブジェクトレベル → 項目レベル → レコードレベル）に沿った多層防御を実現する。

---

## 2. 項目レベルセキュリティ（FLS）

### 2.1 基本原則

**すべての Apex コードにおいて FLS チェックは必須**とする。FLS をバイパスするコードは、明示的なビジネス要件と承認がない限り禁止する。

### 2.2 実装方式の選定基準

| 方式 | 推奨度 | 用途 | 備考 |
|------|-------|------|------|
| `WITH USER_MODE` | **最推奨** | SOQL クエリ全般 | Spring '23 以降。CRUD + FLS を自動チェック |
| `WITH SECURITY_ENFORCED` | 推奨 | 読み取りクエリ | `WITH USER_MODE` が使えない場合のフォールバック |
| `Security.stripInaccessible()` | 推奨 | DML 操作時 | アクセス不可項目を自動除去 |
| `Schema.sObjectType.{Object}.fields.{Field}.isAccessible()` | 条件付き | 個別項目の動的チェック | 動的にオブジェクト/項目を扱う場合のみ |
| `WITHOUT USER_MODE` / `SYSTEM_MODE` | **要承認** | システム処理 | バッチ処理等で明示的に必要な場合のみ。ADR での承認必須 |

### 2.3 FLS バイパスが許可されるケース

以下のケースに限り、`SYSTEM_MODE` の使用を検討できる。ただし、ADR での承認と、コードコメントでの理由明記を必須とする。

| ケース | 理由 | 必要な対応 |
|--------|------|-----------|
| バッチ Apex によるシステム更新 | 実行ユーザーに依存しないデータ整合性の維持 | ADR + コードコメント |
| Platform Event トリガ | Automated Process ユーザーで実行されるため | ADR + コードコメント |
| インテグレーション用ユーザーの処理 | 専用インテグレーションユーザーで実行 | ADR + コードコメント |

## 3. CRUD セキュリティ

### 3.1 基本原則

DML 操作の前に、操作対象オブジェクトに対する CRUD 権限を確認すること。`WITH USER_MODE` を使用している場合は SOQL 側で自動チェックされるが、DML 側は別途確認が必要。

### 3.2 実装パターン

```
【推奨パターン】
SOQL: WITH USER_MODE（CRUD + FLS を自動チェック）
DML: Security.stripInaccessible(AccessType.CREATABLE, records) を実行後に insert

【禁止パターン】
SOQL: クエリに USER_MODE/SECURITY_ENFORCED 指定なし
DML: チェックなしで直接 insert/update/delete
```

## 4. 共有ルール・レコードアクセス

### 4.1 基本原則

- Apex コードは原則として `with sharing` を宣言すること
- `without sharing` の使用は、明示的なビジネス要件がある場合に限定し、コードコメントで理由を記述すること
- `inherited sharing` は、ユーティリティクラスや共通ライブラリクラスで使用すること

### 4.2 選定基準

| 宣言 | 用途 | 使用条件 |
|------|------|---------|
| `with sharing` | 一般的なビジネスロジック | デフォルト。特別な理由がない限りこれを使用 |
| `without sharing` | システム処理、管理者向け機能 | ビジネス要件で共有ルールのバイパスが必要な場合。コードコメント必須 |
| `inherited sharing` | ユーティリティクラス、ヘルパーメソッド | 呼び出し元の共有設定を継承する場合 |

## 5. SOQL インジェクション対策

### 5.1 基本原則

動的 SOQL を使用する場合は、必ずバインド変数またはエスケープ処理を適用すること。

### 5.2 対策方法

| 方式 | 推奨度 | 用途 |
|------|-------|------|
| バインド変数（`:variable`） | **最推奨** | 動的 SOQL の WHERE 句 |
| `String.escapeSingleQuotes()` | 推奨 | バインド変数が使用できない場合 |
| 文字列連結による直接埋め込み | **禁止** | — |

## 6. 機密データの取り扱い

### 6.1 分類と保護レベル

| 分類 | 例 | 保護要件 |
|------|----|---------|
| 個人情報（PII） | 氏名、メールアドレス、電話番号 | FLS による項目レベルアクセス制御 + 暗号化（Shield Platform Encryption 推奨） |
| 機密ビジネスデータ | 売上金額、利益率、契約条件 | FLS + プロファイル/権限セットによるアクセス制御 |
| システム認証情報 | API キー、パスワード、トークン | Named Credential / 保護済みカスタム設定。Apex コード内への埋め込み禁止 |
| 一般データ | 取引先名、商談ステージ | 標準のアクセス制御 |

### 6.2 ログ・デバッグにおける制約

- デバッグログに個人情報を出力しないこと（`System.debug()` で PII を直接出力禁止）
- 外部連携のリクエスト/レスポンスログには、個人情報をマスクしてから保存すること
- 本番環境のデバッグログは最小限に設定し、不要になったら速やかに削除すること

## 7. 権限設計の原則

### 7.1 最小権限の原則

- プロファイルは最小権限で設定し、追加権限は権限セットで付与する
- `Modify All Data` / `View All Data` は管理者プロファイルのみに限定する
- API アクセス権限は、API を利用するユーザー/インテグレーションにのみ付与する

### 7.2 権限セットの設計方針

| 権限セット種別 | 命名規約 | 用途 |
|-------------|---------|------|
| 機能別 | `PS_{機能名}` | 特定の機能へのアクセスを付与（例: `PS_ScoreViewer`） |
| ロール別 | `PS_{ロール名}` | 職務に基づく権限のまとまり（例: `PS_SalesManager`） |
| インテグレーション用 | `PS_Integration_{連携先}` | 外部連携用の権限（例: `PS_Integration_ERP`） |

## 8. 本ドキュメントの参照方法

### サブエージェントへのガイダンス

- **sf-designer**: 設計書に FLS 方式・共有設定・権限設計を記載する際に本方針を参照すること
- **sf-implementer**: Apex 実装時に FLS チェック・`with sharing` 宣言・インジェクション対策を本方針に従って実装すること
- **sf-tester**: テストケースに FLS 違反テスト（権限のないユーザーでのアクセス試行）を含めること
- **sf-code-reviewer**: セキュリティ観点のレビューチェックリストとして本方針を使用すること。`without sharing` や FLS バイパスの使用箇所は重点レビュー対象とすること
