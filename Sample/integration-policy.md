# 外部連携方針（Integration Policy）

> 最終更新: 2026-03-09
> 管理者: {チーム名}
> ステータス: 承認済み

---

## 1. 目的

Salesforce 組織と外部システム間の連携において、セキュリティ・信頼性・保守性を担保するための統一方針を定義する。本ドキュメントは全案件に適用される。

---

## 2. 認証・接続の原則

### 2.1 Named Credential の必須使用

すべての外部コールアウトは **Named Credential** を経由すること。ハードコードされたエンドポイントURL・認証情報のApexコード内への埋め込みは禁止する。

| 項目 | 方針 |
|------|------|
| エンドポイントURL | Named Credential で管理（カスタム設定・カスタムメタデータ型での補完可） |
| 認証情報 | Named Credential の認証プロバイダーで管理 |
| API キー | Named Credential のカスタムヘッダーまたはカスタム設定（保護済み）で管理 |
| 証明書 | Salesforce 証明書管理機能で管理 |

### 2.2 認証方式の選定基準

| 連携先の特性 | 推奨認証方式 | 備考 |
|-------------|------------|------|
| サーバー間連携（バッチ等） | OAuth 2.0 Client Credentials | ユーザーコンテキスト不要の場合 |
| ユーザー操作起点の連携 | OAuth 2.0 Authorization Code | ユーザーの同意が必要な場合 |
| 固定 API キー | API Key（カスタムヘッダー） | 簡易な外部APIの場合 |
| 相互認証が必要 | mTLS + JWT Bearer | 高セキュリティ要件の場合 |

## 3. コールアウト設計の原則

### 3.1 同期 vs 非同期の選定基準

| 条件 | 方式 | 実装パターン |
|------|------|------------|
| レスポンスを即時に画面表示する必要がある | 同期コールアウト | Apex の `HttpRequest` / LWC の `@wire` |
| 処理時間が5秒を超える可能性がある | 非同期（Queueable） | `System.enqueueJob()` でコールアウト実行 |
| 大量レコードの一括送信 | 非同期（Batch） | `Database.Batchable` + `Database.AllowsCallouts` |
| イベント駆動（疎結合にしたい） | Platform Event | Platform Event → Trigger → Queueable でコールアウト |

### 3.2 タイムアウト設定

| 種別 | タイムアウト値 | 備考 |
|------|-------------|------|
| 同期コールアウト（デフォルト） | 10秒 | `req.setTimeout(10000)` |
| 同期コールアウト（最大） | 120秒 | 画面操作時は10秒以内を強く推奨 |
| バッチ処理内のコールアウト | 120秒 | 1トランザクションあたり最大100回 |

## 4. エラーハンドリングとリトライ

### 4.1 リトライ方針

| HTTP ステータス | 対応 | リトライ | 備考 |
|---------------|------|---------|------|
| 2xx | 正常処理 | 不要 | — |
| 400 Bad Request | リクエスト修正 | 不可 | ログ記録 + 業務エラーとして通知 |
| 401 / 403 | 認証エラー | トークンリフレッシュ後に1回 | Named Credential の自動リフレッシュを利用 |
| 404 | リソース不在 | 不可 | ログ記録 + 業務エラーとして処理 |
| 429 | レート制限 | 指数バックオフで最大3回 | `Retry-After` ヘッダーを尊重 |
| 500 / 502 / 503 | サーバーエラー | 指数バックオフで最大3回 | 初回: 1秒、2回目: 2秒、3回目: 4秒 |
| タイムアウト | 接続/読み取りタイムアウト | 最大2回 | 冪等性が保証された API のみ |

### 4.2 リトライの実装パターン

リトライは **Queueable チェーン** で実装する。同一トランザクション内でのループリトライは禁止する。

```
失敗検知 → Integration_Retry__e（Platform Event）発行
  → RetrySubscriber Trigger → RetryQueueable.enqueueJob()
    → 成功: 正常処理
    → 失敗（リトライ上限未到達）: 再度 Platform Event 発行
    → 失敗（リトライ上限到達）: Integration_Error_Log__c 作成 + 管理者通知
```

### 4.3 冪等性の確保

外部APIへの書き込み系リクエストでは、冪等性を確保するために以下を遵守する。

- リクエストに一意な `Idempotency-Key`（Salesforce レコードID または UUID）を付与する
- 連携先が冪等性をサポートしていない場合は、送信前に「送信済みチェック」を実施する
- `Integration_Transaction_Log__c` カスタムオブジェクトで送信履歴を管理する

## 5. データマッピングと変換

### 5.1 マッピング管理方針

| 方式 | 用途 | 採用基準 |
|------|------|---------|
| カスタムメタデータ型 | 項目マッピング定義 | マッピングが頻繁に変更される場合 |
| Apex 定数クラス | 固定的なマッピング | 変更頻度が極めて低い場合 |
| 外部テーブル | 大量のマッピングデータ | 1,000件を超えるマッピングの場合 |

### 5.2 データ変換の原則

- 日時: すべて **UTC** で送受信し、Salesforce 内ではユーザーのタイムゾーンで表示する
- 通貨: `CurrencyIsoCode` を明示的に指定する（マルチ通貨組織の場合）
- 文字コード: UTF-8 を標準とする
- null 値: 連携先の仕様に応じて、null と空文字を明確に区別する

## 6. 監視とログ

### 6.1 ログ記録の必須項目

すべての外部連携において、以下の情報を `Integration_Transaction_Log__c` に記録する。

| 項目 | 内容 | 必須 |
|------|------|------|
| Timestamp__c | 送受信日時（UTC） | ○ |
| Direction__c | Inbound / Outbound | ○ |
| Integration_Name__c | 連携名（INT-001 等） | ○ |
| Endpoint__c | リクエストURL | ○ |
| HTTP_Method__c | GET / POST / PUT / PATCH / DELETE | ○ |
| HTTP_Status__c | レスポンスステータスコード | ○ |
| Request_Body__c | リクエストボディ（個人情報マスク済み） | △（デバッグ時） |
| Response_Body__c | レスポンスボディ（先頭5,000文字まで） | △（エラー時） |
| Error_Message__c | エラーメッセージ | △（エラー時） |
| Related_Record_Id__c | 関連する Salesforce レコードID | △ |
| Retry_Count__c | リトライ回数 | ○ |

### 6.2 アラート通知

| 条件 | 通知先 | 手段 |
|------|--------|------|
| 連続3回以上のエラー | システム管理者 | メール + Slack（#alert-integration） |
| 日次エラー率 5% 超 | システム管理者 + 開発チーム | メール + Slack |
| 認証エラー（401/403） | システム管理者 | メール（即時） |

## 7. 本ドキュメントの参照方法

### サブエージェントへのガイダンス

- **sf-requirements-analyst**: 新しい外部連携が要件に含まれる場合、本ドキュメントの連携方式の選定基準を参照して要件定義書に反映すること
- **sf-designer**: 設計書に連携方式を記載する際、本ドキュメントの方針に準拠すること。新しい連携パターンが必要な場合は ADR を起票すること
- **sf-implementer**: Named Credential の使用、リトライパターン、ログ記録を本方針に従って実装すること
- **sf-code-reviewer**: コールアウト実装が本方針に準拠しているかをレビュー観点に含めること
