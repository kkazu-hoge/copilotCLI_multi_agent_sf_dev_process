# パフォーマンス方針（Performance Policy）

> 最終更新: 2026-03-09
> 管理者: {チーム名}
> ステータス: 承認済み

---

## 1. 目的

Salesforce のガバナ制限を遵守し、大量データ環境でも安定したパフォーマンスを発揮するための設計・実装方針を定義する。本ドキュメントは全案件に適用される。

> **注記**: ガバナ制限の具体値（SOQL 100回、DML 150回等）は `.github/instructions/apex-coding.md` に記載する。本ドキュメントでは設計レベルの方針を定義する。

---

## 2. バルク化の原則

### 2.1 基本方針

すべての Apex コードは **200レコード以上の一括処理** に対応すること。Data Loader や API による一括操作、トリガのバルク実行を常に想定する。

### 2.2 禁止パターン

以下のパターンはすべての Apex コードにおいて禁止する。

| パターン | 理由 | 対処方法 |
|---------|------|---------|
| ループ内 SOQL | ガバナ制限超過（100回/トランザクション） | ループ前に一括取得し Map に格納 |
| ループ内 DML | ガバナ制限超過（150回/トランザクション） | List に蓄積してループ後に一括 DML |
| ループ内コールアウト | ガバナ制限超過（100回/トランザクション） | Queueable チェーンまたは Batch で処理 |
| ループ内 `Messaging.sendEmail()` | ガバナ制限超過（10回/トランザクション） | List に蓄積して一括送信 |
| ハードコードされた ID | 環境間の移植性を破壊 | カスタム設定またはカスタムメタデータ型で管理 |

### 2.3 推奨パターン

| パターン | 用途 | 詳細 |
|---------|------|------|
| Map 化パターン | 関連データの効率的参照 | `Map<Id, SObject>` に一括取得結果を格納 |
| Selector パターン | SOQL の集約 | Selector クラスに SOQL を集約し、再利用性を確保 |
| UnitOfWork パターン | DML の集約 | DML 操作を UnitOfWork に蓄積し、一括コミット |

## 3. SOQL 最適化

### 3.1 クエリ設計の方針

| 方針 | 内容 |
|------|------|
| 選択的クエリ | WHERE 句にインデックス付き項目を含め、大量データでもパフォーマンスを維持する |
| 必要項目のみ取得 | `SELECT *` 相当のクエリは禁止。必要な項目のみを明示的に指定する |
| 親子関係の活用 | 子→親の参照はドット表記（`Account.Name`）、親→子はサブクエリで1回のクエリに集約する |
| 集計関数の活用 | COUNT/SUM/MAX 等はクエリレベルで実行し、Apex 側でのループ集計を避ける |

### 3.2 インデックス戦略

| 項目の特性 | インデックスの推奨 | 備考 |
|-----------|-----------------|------|
| 外部 ID 項目 | 自動インデックスあり | 外部連携のキー項目に使用 |
| 標準項目（Name, CreatedDate 等） | 自動インデックスあり | WHERE 句で積極的に活用 |
| カスタム項目（選択性の高いもの） | カスタムインデックス申請を検討 | ユニーク項目、外部 ID 以外で WHERE 句に頻用する場合 |
| 数式項目、テキストエリア | インデックス不可 | WHERE 句での使用を避ける |

## 4. 大量データ処理の設計方針

### 4.1 処理方式の選定基準

| データ量 | 処理方式 | 備考 |
|---------|---------|------|
| ～200件 | 同期処理（トリガ / LWC） | 標準的なトランザクション内で完結 |
| 200～50,000件 | Queueable Apex | チェーン実行で分割処理 |
| 50,000件～ | Batch Apex | `Database.Batchable` で200件ずつ処理 |
| 100万件～ | Bulk API 2.0 + Batch Apex | Data Loader / 外部連携で一括投入後、バッチで後処理 |

### 4.2 Batch Apex の設計方針

| 項目 | 方針 |
|------|------|
| scope サイズ | デフォルト200。コールアウトを含む場合は scope を小さくする（例: 50） |
| エラーハンドリング | `Database.insert(records, false)` で部分成功を許容し、エラーレコードをログに記録 |
| ステータス管理 | `AsyncApexJob` の監視 + カスタムオブジェクトでの進捗記録 |
| 並行実行 | 同一 Batch クラスの並行実行は原則禁止（`flex queue` の制約を考慮） |

## 5. LWC パフォーマンス方針

### 5.1 データアクセスの方針

| 方式 | 推奨度 | 用途 |
|------|-------|------|
| Lightning Data Service (LDS) | **最推奨** | 単一レコードの CRUD。キャッシュが自動的に活用される |
| `@wire` + Apex | 推奨 | 複数レコードの取得。キャッシュ可能 |
| Imperative Apex（`@api` なし） | 条件付き | ユーザーアクション起点のデータ取得・更新 |

### 5.2 描画パフォーマンスの方針

| 方針 | 内容 |
|------|------|
| 遅延ロード | 初期表示に不要なデータは、タブ切り替えやスクロール時に取得する |
| ページネーション | 一覧表示は1ページあたり最大50件とし、OFFSET ではなくカーソルベースのページネーションを使用する |
| デバウンス | 検索入力等の頻繁なイベントには 300ms 以上のデバウンスを適用する |
| コンポーネント分割 | 大規模コンポーネントは子コンポーネントに分割し、再レンダリング範囲を最小化する |

## 6. テスト・監視に関する方針

### 6.1 パフォーマンステストの要件

| テスト種別 | 対象 | 基準 |
|-----------|------|------|
| バルクテスト | すべてのトリガ・バッチ処理 | 200レコード以上での正常動作を確認 |
| ガバナ制限テスト | 複雑なトランザクション | `Limits` クラスで SOQL/DML 回数を検証 |
| 大量データテスト | バッチ処理 | 想定最大データ量の 1.5 倍で実行し、処理時間を記録 |

### 6.2 本番環境の監視

| 監視項目 | 閾値 | アクション |
|---------|------|----------|
| SOQL クエリ回数 | トランザクションあたり80回以上 | 警告ログ出力 + 最適化検討 |
| CPU 時間 | トランザクションあたり8,000ms以上 | 警告ログ出力 + 非同期化検討 |
| ヒープサイズ | トランザクションあたり 9MB 以上（同期時） | 警告ログ出力 + データ分割検討 |
| Batch Apex 実行時間 | 1ジョブあたり2時間以上 | scope サイズの見直し |

## 7. 本ドキュメントの参照方法

### サブエージェントへのガイダンス

- **sf-designer**: 設計書にデータ量の見積りと処理方式（同期/非同期/バッチ）の選定理由を記載すること
- **sf-implementer**: バルク化パターン、SOQL 最適化、LWC データアクセス方式を本方針に従って実装すること
- **sf-tester**: バルクテストとガバナ制限テストを必ず実施すること
- **sf-code-reviewer**: ループ内 SOQL/DML、ハードコード ID の検出を重点レビュー項目とすること
